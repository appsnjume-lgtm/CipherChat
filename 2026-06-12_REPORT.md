# CipherChat Audit Report - 2026-06-12

## 1. Executive Summary

This report provides a comprehensive security, performance, and quality audit of the CipherChat application. CipherChat is a Flutter-based encrypted messaging platform using Supabase as its backend. 

Overall, the project demonstrates a high level of security awareness, particularly with its End-to-End Encryption (E2EE) implementation and the recent application of "Pre-production hardening" migrations. Most of the critical architectural and performance bottlenecks identified in previous internal reviews (such as the N+1 chat hydration pattern and realtime subscription bloat) have been addressed via consolidated RPCs and inbox-scoped subscriptions.

However, several areas for improvement remain, specifically regarding UX resilience during profile loading, accessibility support, and database query optimization for large groups.

## 3. High Severity Issues

### 3.1. Profile Setup Gate Race Condition
*   **Severity:** High
*   **Description:** The `AuthGateScreen` renders the `AuthScreen` (profile setup) if a user is authenticated but their profile is not yet loaded or cached.
*   **Affected File(s):** `lib/features/auth/presentation/providers/auth_provider.dart`, `lib/features/auth/presentation/screens/auth_gate_screen.dart`
*   **Why it is risky:** Users with existing profiles on the server will briefly see the "Finish your profile" UI upon every cold start if the local cache is empty or cleared. On slow networks, this leads to a poor UX where users think their account is reset or they are forced to re-enter details.
*   **Recommended Fix:** Update `needsProfileSetup` to check `!isProfileLoading`. Show a loading skeleton while the remote profile fetch is in progress.

## 4. Medium Severity Issues

### 4.1. Message Receipt Over-fetching in Groups
*   **Severity:** Medium
*   **Description:** The `fetchMessages` repository method selects all receipts for every message: `select('*, message_receipts(*)')`.
*   **Affected File(s):** `lib/features/chat/data/repositories/chat_repository.dart`
*   **Why it is risky:** In a group with 100 members, a single page of 30 messages will return up to 3,000 receipt rows. This increases payload size, memory usage, and JSON parsing time on the client.
*   **Recommended Fix:** Only fetch receipts for the current user by default. Implement a separate lazy-load for full group read status.

### 4.2. Lack of Pagination in `get_chat_inbox`
*   **Severity:** Medium
*   **Description:** The `get_chat_inbox` RPC returns all chats for a user without limit or offset.
*   **Affected File(s):** `supabase/migrations/20260416_repair_plan_sql_rollup.sql`
*   **Why it is risky:** Performance will degrade linearly as a user's chat count grows (e.g., 200+ chats).
*   **Recommended Fix:** Add `p_limit` and `p_offset` parameters to the `get_chat_inbox` function.

### 4.3. Missing Semantics for Accessibility
*   **Severity:** Medium
*   **Description:** Critical UI components like `AppAvatar`, `ChatTile`, and `MessageBubble` lack `Semantics` widgets or content descriptions.
*   **Affected File(s):** `lib/common/widgets/app_avatar.dart`, `lib/features/chat/presentation/widgets/chat_tile.dart`, `lib/features/chat/presentation/widgets/message_bubble.dart`
*   **Why it is risky:** Users relying on TalkBack or Screen Readers will have a degraded experience, hearing "Unlabelled button" or just the raw text without context (e.g., who sent the message).
*   **Recommended Fix:** Wrap avatars and tiles in `Semantics` with appropriate labels.

## 5. Low Severity Issues

### 5.1. Redundant `READ_EXTERNAL_STORAGE` Permission
*   **Severity:** Low
*   **Description:** The manifest includes `READ_EXTERNAL_STORAGE` with `maxSdkVersion="32"`.
*   **Affected File(s):** `android/app/src/main/AndroidManifest.xml`
*   **Why it is risky:** The app uses `image_picker` and `file_picker` which handle permissions internally. Extra permissions can slightly decrease user trust.
*   **Recommended Fix:** Verify if `file_picker` requires it for specific edge cases; otherwise, remove to minimize permission footprint.

## 6. Security Findings

*   **E2EE Implementation:** The use of `cryptography` and `encrypt` packages for payload encryption is correctly implemented. The `secure_chat_service.dart` ensures keys are unwrapped only for the authorized user.
*   **RLS Hardening:** Recent migrations have successfully restricted `profiles`, `stickers`, and `group-images`.
*   **Hardcoded Secrets:** No live Supabase secrets were found in the source code; the app correctly uses `dart-define` with a `config.json` fallback (which is not bundled in assets).
*   **Component Export:** `MainActivity` is exported (required for launcher), and deep link filters are correctly scoped to `cipherchat://auth-callback`.

---

## 7. Database Findings (Supabase)

*   **Audit of `/supabase/migrations`:**
    *   `20260416_repair_plan_sql_rollup.sql`: Correctly implements idempotent fixes for group images, stickers, and the inbox RPC.
    *   `20260413_harden_grid_breach...`: Successfully moves game logic (turn validation, timeouts) to server-side PL/pgSQL.
*   **RLS Policies:** Most tables have robust policies. One observation: `profiles` `SELECT` is restricted to `auth.uid() = id`. This is very secure but forces all member lookups through `security definer` functions. Ensure developers are aware of this pattern.
*   **Indexes:** Good use of GIN indexes for trigram search on usernames and titles. B-tree indexes exist for most foreign key lookups used in `get_chat_inbox`.

---

## 8. Performance Findings

*   **Startup Speed:** The previous 8-second startup issue has been mitigated by the `get_chat_inbox` RPC, which reduced network round trips from O(N) to O(1).
*   **Realtime:** Transitioning from per-chat channels to a single `message_receipts` inbox listener has significantly reduced the WebSocket overhead at startup.
*   **Media Handling:** FFmpeg is a heavy dependency (`ffmpeg_kit_flutter_new`). While necessary for Sticker WebP conversion, it increases the binary size significantly.

---

## 9. Accessibility Findings

*   **TalkBack:** Navigation in the chat list is functional but lacks descriptive context for unread counts and online status.
*   **Touch Targets:** Most buttons meet the 48x48dp recommendation, but some "GX" theme elements use custom painters that may not accurately report touch bounds to accessibility services.

## 11. Dependency Findings

*   **FFmpeg Kit:** Using the "Full" or large variant of FFmpeg may be overkill if only WebP conversion is needed.
*   **Riverpod:** Currently using `3.3.1` (which is a very recent/beta version if following standard semver, likely using the latest dev versions). Ensure stability matches project requirements.

---

## 12. Architecture Findings

*   **Pattern:** Excellent use of Clean Architecture. Repositories are well-separated from Presentation logic.
*   **State Management:** Riverpod is used effectively to handle complex states like the Grid Breach game and real-time messaging.
*   **Scalability:** The move towards RPC-based data fetching makes the app much more scalable than the initial "Direct-from-PostgREST" approach.

---

## 13. Quick Wins

1.  Add `Semantics` to `AppAvatar` (e.g., `label: "Avatar of $username"`).
2.  Update `needsProfileSetup` in `auth_provider.dart` to prevent the "Auth Screen Flash" during cold starts.
3.  Optimize the `fetchMessages` query to only select `message_receipts` for the current user.

---

1.  **High:** Fix `needsProfileSetup` logic to improve startup UX.

3.  **Medium:** Implement pagination for `get_chat_inbox` RPC.
4.  **Medium:** Optimize message receipt fetching in `fetchMessages`.
